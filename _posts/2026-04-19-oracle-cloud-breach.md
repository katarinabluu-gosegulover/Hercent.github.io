- **영향 테넌트:** 140,000+
- **탈취 레코드:** 6,000,000+
- **서버 마지막 업데이트:** 2014년

## Key Metrics
- **탈취된 인증 레코드:** 6M
- **영향받은 클라우드 테넌트:** 140K+
- **취약점 공개 후 패치 지연:** 4년
- **Oracle에 요구한 몸값(Monero):** $200M

## 클라우드 공급망의 아킬레스건

_사건 개요_

2025년 3월 20일, 'rose87168'이라는 별칭의 위협 행위자가 해킹 포럼 BreachForums에 오라클 클라우드의 SSO(Single Sign-On)와 LDAP 시스템에서 탈취한 약 **600만 건의 레코드**를 판매한다는 글을 올렸다. 피해 기업 목록에는 140,000개 이상의 기업 도메인이 포함됐다.

탈취된 데이터는 Java KeyStore(JKS) 파일, 암호화된 SSO 패스워드, 키 파일, 엔터프라이즈 매니저 JPS 키 등 인증 체계의 핵심 요소들을 포함했다. 이 데이터가 크래킹 또는 활용될 경우, 오라클 클라우드를 사용하는 수천 개 기업의 시스템이 2차 침해에 노출될 수 있다.

> **⚡ Oracle의 공식 입장 vs 보안 연구자들의 주장**
> Oracle은 공식적으로 "Oracle Cloud 침해는 없었다"고 부인했다. 그러나 CloudSEK, SOCRadar 등의 보안 연구자들은 샘플 데이터를 실제 고객들에게 확인받았고, 공격 당한 서버의 Internet Archive 기록을 통해 취약 버전의 Oracle Fusion Middleware 11g가 2025년 2월까지 운용되고 있었음을 증명했다.
> Oracle의 부인은 피해 기업들이 자격증명을 교체하지 않을 위험을 높여 2차 피해 가능성을 키웠다는 비판을 받았다.

## CVE-2021-35587: 4년 전 경고의 귀환

_취약점 분석_

**CVE-2021-35587 — Oracle Fusion Middleware / Access Manager**
- **CVSS Score:** 9.8 (Critical)
- **공개일:** 2022년 1월 (Oracle CPU 패치)
- **영향 제품:** Oracle Access Manager (OAM) 11.1.2.3
- **공격 벡터:** 네트워크 접근 가능, 인증 불필요
- **공격 결과:** OAM 완전 장악 (Takeover)
- **CISA KEV:** 2022년 말 CISA Known Exploited 목록에 추가됨
- **Oracle 클라우드 서버:** 2014년 이후 업데이트 없이 운영 (Internet Archive 증거)

이 취약점은 HTTP 요청을 통해 Oracle Access Manager 인스턴스를 인증 없이 완전 장악할 수 있게 한다. 2022년 Oracle이 패치를 발표하고 CISA가 Known Exploited Vulnerabilities(KEV) 목록에 추가할 정도로 위험성이 알려졌음에도, 오라클 클라우드의 프로덕션 SSO 엔드포인트는 패치 없이 수년간 운용되고 있었다.

> **⚠ 패치 관리의 실패**
> attack된 login.us2.oraclecloud.com 서버가 Oracle Fusion Middleware 11g를 실행하고 있었으며, 인터넷 아카이브 기록에 따르면 마지막으로 확인된 버전 업데이트는 2014년이었다. 클라우드 서비스 제공자가 자사 프로덕션 인프라에서 11년 된 소프트웨어를 운용하고 있었다는 사실은 업계에 큰 충격을 주었다.

## 공격자는 어떻게 접근했나

_공격 흐름_

1. **STEP 01 - 취약한 SSO 엔드포인트 식별**
   공격자(rose87168)가 `login.us2.oraclecloud.com` 및 각 리전별 `login.(region).oraclecloud.com` 엔드포인트를 스캔. Oracle Fusion Middleware 11g 실행 확인.
1. **STEP 02 - CVE-2021-35587 익스플로잇**
   비인증 HTTP 요청으로 `Oracle Access Manager` 인스턴스 장악. 관리자 세션 토큰 획득. 이후 인증 없이 SSO 및 LDAP 백엔드 시스템 접근 가능.
1. **STEP 03 - 인증 데이터 대규모 추출**
   140,000+ 테넌트의 LDAP/SSO 레코드, `.jks` 키스토어 파일, 암호화된 패스워드 해시, SAML/OIDC 설정 파일, Enterprise Manager JPS 키 등 대량 추출.
1. **STEP 04 - Oracle에 몸값 요구 ($200M Monero)**
   Oracle에 직접 10만 Monero(약 $200M 상당) 요구. Oracle이 거부하자 BreachForums에 데이터 판매 공고.
1. **STEP 05 - 공개 판매 및 탈취 증거 공개**
   3개의 샘플 파일(LDAP, Company List, Database) 공개. 영향받은 기업들에게 데이터 확인 후 제거 비용 결제 요구. X(트위터)에서 Oracle 관련 계정 팔로우하며 공개 압박.

| 데이터 유형 | 설명 | 잠재 악용 위험 |

| LDAP 패스워드 해시 | 기업 디렉터리 서비스의 사용자 인증 해시. 크래킹 성공 시 광범위한 내부 시스템 접근 가능. | 치명적 |

| SAML/OIDC 설정 | 연동된 외부 서비스 인증 설정. 공급망 2차 침해 경로 제공. | 높음 |

> **🔴 공급망 연쇄 위험**
> JKS 키파일과 SSO 패스워드가 성공적으로 복호화될 경우, 단순히 Oracle Cloud 접근에 그치지 않는다. 이 자격증명들은 Oracle Cloud에 연결된 수십~수백 개의 외부 서비스와 API에 대한 접근도 허용할 수 있다. 이른바 "공급망 연쇄 침해"가 가능해진다.

## 왜 이 사고가 발생했나

_근본 원인 분석_

### Oracle의 실패 요인

- 4년 전 공개된 치명적 취약점을 프로덕션 환경에 패치하지 않음
- 2014년 이후 업데이트되지 않은 레거시 Fusion Middleware 11g 운용
- 침해 발생 후 부인으로 기업들의 대응 시간 낭비 유발
- 클라우드 고객 인프라 취약점 관리 체계 부재

### 보안 연구자들의 기여

- CloudSEK이 샘플 데이터를 실제 고객들에게 검증받아 침해 사실 확인
- Internet Archive를 통해 취약 서버 버전 공개적으로 증명
- SOCRadar가 10,000건 샘플 분석 결과 공유
- 영향 기업들에게 즉각적인 자격증명 로테이션 권고 전파

## 무엇이 문제였나

_미흡한 점 & 교훈_

- **패치 관리 실패 - CISA KEV 등재 취약점도 미적용**: CVE-2021-35587은 2022년 말 CISA의 "Known Exploited Vulnerabilities" 목록에 공식 등재됐다. 이는 미국 연방기관의 경우 패치 의무가 부여되는 최고 경계 수준임에도, Oracle의 자사 클라우드 인프라에 적용되지 않았다.
- **레거시 시스템 방치 - 11년 된 소프트웨어의 프로덕션 운용**: 2014년 이후 업데이트 없이 운용된 Oracle Fusion Middleware 11g. 클라우드 서비스 제공자가 자사 핵심 인증 인프라를 이 상태로 방치했다는 것은 소프트웨어 수명주기 관리의 심각한 실패를 의미한다.
- **투명성 부재 - 부인(denial)이 추가 피해 야기**: Oracle의 즉각적인 부인은 140,000개 이상의 기업이 자격증명을 즉시 교체하는 조치를 취하지 않도록 유도했다. 이 기간 동안 공격자가 추가 데이터를 악용했을 가능성이 있다.
- **클라우드 보안 가정 - "클라우드 = 안전"이라는 잘못된 믿음**: 많은 기업들이 클라우드 서비스 제공자(CSP)가 자체 인프라 보안을 책임진다고 암묵적으로 가정했다. 이번 사건은 CSP의 인프라 자체가 취약점을 가질 수 있으며, 고객 기업들도 독립적인 보안 검증이 필요함을 일깨웠다.

## 피해 기업이 즉시 취해야 할 조치

_대응 방안 및 권고_

- **즉각적인 자격증명 로테이션**: Oracle LDAP 및 SSO 패스워드 전체 교체. 테넌트 관리자 계정 최우선. 교체 후 비정상 로그인 집중 모니터링.
- **MFA 강제 적용**: 모든 Oracle Cloud 접근 포인트에 멀티팩터 인증 즉시 의무화. 기존 SSO 세션 강제 무효화.
- **키 및 인증서 재생성**: 탈취된 JKS 파일과 연관된 모든 인증서, SAML/OIDC 시크릿, LDAP 설정 교체.
- **접근 로그 감사**: 2025년 1월 이후 Oracle Cloud 인증 로그 전수 검토. 비정상 접근 패턴, 새로운 API 키 발급, 권한 변경 이력 확인.
- **공급망 연결 서비스 점검**: Oracle Cloud SSO와 연동된 외부 시스템 목록 파악 및 연동 토큰 일괄 폐기·재발급.
- **클라우드 보안 아키텍처 재검토**: CSP의 보안 패치 적용 여부를 독립적으로 검증하는 체계 구축. 클라우드 소프트웨어 버전 공개 모니터링.

> **💡 클라우드 보안에 대한 패러다임 전환**
> 이번 사건은 클라우드 보안의 "공동 책임 모델(Shared Responsibility Model)"에 대한 재정의를 요구한다. Oracle 같은 대형 CSP라도 자사 인프라의 패치 관리와 레거시 시스템 운용에 실패할 수 있다. 고객 기업들은 CSP의 인프라 보안을 맹목적으로 신뢰하기보다, 연결 자격증명의 독립적 관리와 정기적 로테이션을 내부 정책으로 정착시켜야 한다.
