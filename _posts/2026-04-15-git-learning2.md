## Git Command Learning (원격)

`git clone` 은 원격 저장소의 복사본을 _로컬_에 생성할때 사용하는 명령어(ex: github에서 가져올 때)

---

우리의 로컬 저장소에 o/main라고하는 새 브랜치가 생기는데, 이런 종류의 브랜치는 **_원격_브랜치** 라고 불린다. 원격 브랜치는 특정한 목적을 제공하기 때문에 특별한 속성들이 있다.

원격 브랜치는 원격 저장소의 **상태**를 반영합니다(가장 최근 원격 원격저장소와 작업을 했을때를 기준으로). 원격 브랜치는 로컬에서의 작업과 공개적으로 되고있는 작업의 차이를 이해하는데 도와준다. *다른 사람들과 작업을 공유하기전에 반드시해야할 과정*

 원격 브랜치 또한 (필수적인) 이름짓기 규약이 있다.  다음의 형식으로 나타납니다:

`<remote name>/<branch name>`
이런 이유로, 만약 o/main라는 이름의 브랜치를 보게되면, 브랜치의 이름은 main이고 원격 저장소의 이름은 o인겁니다.

대부분의 개발자들은 자신의 주 원격 저장소를 o가 아닌 origin이라고 짓는다. 사실 보통 다 이렇게 쓰기 때문에 git은 저장소를 git clone하게 되면 원격 저장소의 이름을 origin이라고 자동으로 설정해놓는다.

---

git fetch는 두가지의 중요한 단계를 수행함.

원격 저장소에는 있지만 로컬에는 없는 커밋들을 다운로드 받는다. 그리고...
우리의 원격 브랜치가 가리키는곳을 업데이트한다 (예를들어, origin/main).
`git fetch` 는 본질적으로 _로컬_에서 나타내는 원격 저장소의 상태를 실제 원격 저장소의 (현재)상태와 동기화한다.

`git fetch` 는 일반적으로 원격 저장소와 인터넷을 통해 접근한다.(http:// 또는 git:// 와같은 프로토콜로)

---

`git pull` 은 본질적으로 `git fetch` 후에 내려받은 브랜치를 병합(merge) 하는 과정의 단축 명령어.

---

`git push` 는 변경점을 정한 원격저장소에 업로드하고 그 원격 저장소가 새 커밋들을 합치고 갱신하게 한다. `git push` 가 끝나고 나면, 동료들은 원격저장소에서 우리의 작업을 내려받을수 있게된다.

---

아무 임의의 브랜치를 o/main를 추적하게 만들 수 있다. 이렇게 하면 이 브랜치 또한 내재된 push,merge 목적지를 main로 할 것이다.
totallyNotMain라는 브랜치에서 `git push` 를 수행해서 원격 저장소의 브랜치 main로 작업을 push할 수 있다.

이 속성을 설정하는데에는 두가지 방법이 있는데, 첫 번째는 지정한 원격 브랜치를 참조해서 새로운 브랜치를 생성하여 checkout 하는 방법이고, 다음을 실행하면

`git checkout -b totallyNotMain o/main`

totallyNotMain라는 이름의 새 브랜치를 생성하고 o/main를 추적하게 설정한다.

브랜치에 원격 추적 설정을 하는 또 다른 방법으로는 간단하게 `git branch -u `옵션을 사용하는 방법이 있다. 다음을 실행하면

`git branch -u o/main foo`

가 foo 브랜치가 o/main를 추적하도록 설정한다. 만약 foo가 현재 작업하고 있는 브랜치라면 생략해도 됩니다:

`git branch -u o/main`

---

`git push origin main`

해석: 내 저장소에 있는 "main"라는 이름의 브랜치로 가서 모든 커밋들을 수집하고, 그다음 "origin"의 "main"브랜치로 가서 이 브랜치에 부족한 커밋들을 채워 넣고 완료 되면 알려준다.

main를 "place"인자로 지정해서 우리가 git에게 어디서부터 커밋이 오는지, 그리고 어디로 커밋이 가야하는지 알려준다. 두 저장소간에 동기화 작업을 할 "장소"를 지정해 주는것이라고 볼 수 있다.

`git push origin <source>:<destination>` 를 이용하여 더 자세히 지정가능.
