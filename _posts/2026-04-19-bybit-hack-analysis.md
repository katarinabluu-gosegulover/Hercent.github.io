## 무슨 일이 있었나

_사건 개요_

2025년 2월 21일 오후 12시 30분(UTC), 두바이 기반 암호화폐 거래소 **Bybit**은 정기 콜드-투-웜 지갑 이체 중 이상 거래를 감지했다. 401,347 ETH, 당시 시가 약 **15억 달러**가 단 한 번의 트랜잭션으로 공격자 지갑으로 유출됐다. 이는 암호화폐 역사상 단일 도난 사건 최대 기록이다.

더욱 충격적인 점은 이 공격이 스마트 컨트랙트 취약점이나 개인키 탈취 방식이 아니었다는 것이다. 공격자는 Bybit이 사용하던 멀티시그 솔루션 **Safe{Wallet}**의 공급망을 침해했고, 서명자들이 보는 UI 자체를 조작하여 그들이 자신도 모르게 악의적인 트랜잭션에 서명하도록 유도했다.

> **핵심 포인트**
> Bybit의 인프라는 침해되지 않았다. 공격은 Bybit이 신뢰하던 제3자 서비스 Safe{Wallet}의 개발자 머신을 통해 이루어졌다. 3명의 서명자 모두 정상적인 UI를 보며 서명했지만, 내부적으로는 컨트랙트 제어권이 공격자에게 넘어가는 트랜잭션이었다.

## 17일간의 정밀 작전

_공격 타임라인_

- **2025-02-04 - Safe{Wallet} 개발자 머신 침해**: 소셜 엔지니어링(Contagious Interview 기법 추정)으로 macOS 개발자 워크스테이션 감염. 최초 악성 활동 탐지.
- **2025-02-05 - AWS 인프라 접근**: 탈취한 개발자 AWS 액세스 토큰으로 Safe{Wallet}의 AWS S3 및 CloudFront 인프라에 접근 성공.
- **2025-02-19 15:29 UTC - 악성 JavaScript 주입**: app.safe.global의 정상 JS 파일을 악성 코드로 교체. 코드는 Bybit의 콜드 지갑 주소를 감지하도록 설계됨. 일반 사용자에게는 정상 동작.
- **2025-02-20 - 위장 도메인 등록**: Lazarus Group이 bybit-assessment[.]com 도메인 등록. 추가 피싱 인프라 준비.
- **2025-02-21 14:13 UTC - 공격 실행 — $1.5B 탈취**: Bybit 서명자 3명이 Safe{Wallet} UI에서 정상 이체처럼 보이는 트랜잭션을 승인. 실제로는 멀티시그 컨트랙트 소유권이 공격자에게 이전. 즉시 401,347 ETH 출금.
- **2025-02-21 (공격 직후) - 증거 인멸**: 악성 JS를 Safe{Wallet} 인프라에서 즉시 제거. 동일 벡터 재사용 가능성 유지 목적으로 판단됨.
- **2025-03-20 - 자금 세탁 86% 완료**: 탈취 ETH의 86.29%를 BTC로 전환 후 믹서, DEX, P2P 플랫폼을 통해 세탁. eXch가 Bybit 요청에도 스왑 허용 논란.

## 공격은 어떻게 작동했나

_기술적 분석_

### // 악성 JavaScript 로직

1. **01**: 서명자가 Safe{Wallet} UI(app.safe.global) 접속 → 악성 JS가 요청 가로채기
1. **02**: 조건 확인: 현재 요청 주소가 Bybit 인증 서명자인가? → YES
1. **03**: 원본 트랜잭션 데이터(정상 이체) 캐싱 후 서명 대상 데이터 교체: delegatecall을 통해 컨트랙트 소유권 이전 코드로 변경
1. **04**: 서명자는 UI에서 정상 이체 화면을 확인하고 하드웨어 지갑으로 승인
1. **05**: 서명 완료 후 악성 JS가 원본 트랜잭션 데이터로 복원 (단, 변조된 서명 유지) → 다음 서명자에게 전달
1. **06**: 3/3 서명 수집 완료 → 블록체인에 악성 트랜잭션 제출 → 멀티시그 컨트랙트 소유권 이전 → 즉시 ETH 인출

### // EIP-712 서명의 한계

EIP-712는 구조화된 데이터의 서명 표준이지만, 이번 공격처럼 **중첩된 delegatecall 연산**이 포함된 복잡한 스마트 컨트랙트 작업에서는 서명자가 실제로 무엇을 승인하는지 사람이 읽을 수 있는 형태로 표현하지 못한다. Safe{Wallet}의 다중 계층 트랜잭션 구조에서 하드웨어 지갑 화면에 표시되는 정보와 실제 온체인 실행 내용 사이에 **의미론적 단절**이 발생했다.

> **⚠ 핵심 취약점**
> 멀티시그 보안 모델의 근본적 가정은 "서명자들이 동일한 신뢰할 수 있는 인터페이스를 통해 트랜잭션을 검증한다"는 것이다. 이번 공격은 인터페이스 자체를 오염시킴으로써 서명자 수를 아무리 늘려도 무의미함을 증명했다.

## 왜 이 공격이 성공했나

_근본 원인 분석_

### 공급망 신뢰 과잉

Bybit은 Safe{Wallet}의 JS 파일 무결성을 별도로 검증하지 않았다. CDN/S3를 통해 서빙되는 제3자 스크립트를 완전히 신뢰하는 구조였다.

### Blind Signing 문제

서명자들이 트랜잭션 내용을 UI 화면 이외의 방법으로 독립 검증하지 않았다. 하드웨어 지갑 화면에는 해석 불가능한 raw data만 표시됐다.

### 개발자 보안 부재

Safe{Wallet} 개발자 머신이 소셜 엔지니어링으로 침해되었고, AWS 자격증명이 탈취됐다. 개발자 계정 접근에 대한 추가 인증이 없었다.

### 대규모 이체 지연 없음

15억 달러 규모의 이체가 시간 지연 없이 즉시 실행됐다. Time-lock 메커니즘이 있었다면 이상 감지 시간을 확보할 수 있었다.

## 사후 대응과 개선 방향

_대응 및 교훈_

### // Bybit의 즉각 대응

- 이상 감지 즉시 트위터 공개 발표, 사용자 안심 및 투명한 소통
- Sygnia, Verichains 포렌식 조사 의뢰 및 결과 전문 공개 (업계 투명성 벤치마크 제시)
- 브릿지 론과 내부 자금으로 피해액 전액 커버, 사용자 손실 없음
- FBI 신고 및 도난 자금 추적 협력

### // Safe{Wallet}의 대응

- 전체 인프라 재구축, 자격증명 로테이션, 공격 벡터 완전 제거
- 이더리움 메인넷 단계적 재가동, Ledger 네이티브 통합 임시 비활성화
- 트랜잭션 해시·데이터·서명 추가 검증 강화

### // 업계가 도입해야 할 대책

- **프리-사이닝 시뮬레이션**: 트랜잭션 실행 전 실제 결과를 블록체인 시뮬레이터로 미리 검증
- **UI 독립 트랜잭션 검증**: raw transaction data를 UI 외부 독립 경로로 확인 (오프체인 검증)
- **Time-lock 적용**: 대규모 이체에 최소 24-48시간 지연 시간 설정
- **SRI 체크섬**: CDN 서빙 JS 파일에 Subresource Integrity 해시 강제
- **개발자 머신 하드닝**: 코드 배포 권한 계정에 phishing-resistant MFA 강제 적용
- **최소권한 원칙**: 개발자 1명의 침해가 프로덕션에 직접 영향 미치지 않도록 접근 계층화

> **ℹ 업계에 남긴 교훈**
> 이 사건은 "멀티시그 = 안전"이라는 공식이 서명 인터페이스 신뢰성을 전제함을 증명했다. Chainlink, Aave, StarkNet 등 다수 고가치 프로토콜이 Safe{Wallet}을 사용 중이었으며, 공격자가 Bybit만을 타깃으로 한 이번 코드가 더 많은 조직을 노렸다면 피해는 수십 배에 달할 수도 있었다.
