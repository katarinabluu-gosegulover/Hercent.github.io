## 논문 정보

| 항목 | 내용 |
| --- | --- |
| 논문 제목 | *Inconsistent, Incomplete, and Insecure: A Survey of Account Security Interfaces* |
| 학회 | USENIX Security Symposium 2026 |
| 저자 | Arkaprabha Bhattacharya, Alaa Daffalla, Kevin Lee, Rosanna Bellini, Nicola Dell, Thomas Ristenpart |
| 내가 이 논문을 고른 이유 | 우리가 실제로 쓰는 로그인 기록, 활성 세션, 보안 알림 화면이 얼마나 믿을 만한지 다뤘기 때문이다. 주제가 실생활과 맞닿아 있고, 실측과 공격 재현이 함께 들어 있어 리뷰 가치가 높다고 판단했다. |

## 한눈에 보기

- 조사 대상: TRANCO 기준 상위 100개 웹 서비스
- ASI 발견: 71개 서비스, 총 200개 인터페이스
- ASI 자체가 없는 서비스: 29개
- 스푸핑에 취약한 서비스: 41개, 전체의 67.2%
- 핵심 메시지: 사용자가 스스로 계정 침해를 감지하도록 돕는 인터페이스조차 충분히 신뢰하기 어렵다

## 왜 이 논문이 나왔나

이 논문이 다루는 핵심 문제는 단순한 로그인 보안이 아니다. 저자들이 주목한 건 **사용자가 이미 계정 접근을 허용해버렸거나, 가까운 사람에게 계정이 남용되는 상황에서 사용자가 침해를 스스로 알아차릴 수 있느냐**는 문제다. 가족, 연인, 직장 동료처럼 물리적·심리적으로 가까운 사람이 계정에 접근하는 경우, 피해자는 자신의 계정이 악용되고 있다는 사실조차 뒤늦게 알거나 끝내 알아차리지 못할 수 있다.

이때 사용자가 거의 유일하게 의지할 수 있는 도구가 Account Security Interface, 즉 ASI다. 최근 로그인 기기, 활성 세션, 보안 알림, 로그인 기록 같은 메뉴가 여기에 포함된다. 문제는 이 인터페이스가 실제로 침해 탐지에 도움이 되는 수준으로 설계되어 있는지 아무도 체계적으로 확인하지 않았다는 점이다.

기존 연구도 전혀 없었던 것은 아니다. Daffalla et al. (2023)은 Google, Apple, Facebook, WhatsApp 네 개 서비스를 분석했다. 하지만 서비스 수가 너무 적어 웹 전반으로 일반화하기 어려웠고, 대부분의 기존 논의도 "알림을 더 잘 보낼 방법"에 집중되어 있었다. 반면 이 논문은 **ASI 자체의 품질과 보안성**을 본격적으로 문제 삼는다.

## 이 논문이 새롭게 한 일

저자들의 핵심 아이디어는 단순하다. **100개 서비스를 직접 조사하고, 그 인터페이스를 실제 공격 시나리오에 넣어 흔들어보자**는 것이다. 여기서 중요한 건 두 가지다.

- 첫째, 특정 몇 개 사례가 아니라 대규모 서비스 집합을 직접 조사했다.
- 둘째, UI가 보여주는 정보가 실제로 얼마나 속기 쉬운지 스푸핑 공격으로 검증했다.

즉, 이 논문은 "어떤 서비스에 ASI가 있다"를 넘어서, **그 ASI가 실제 침해 탐지 도구로서 쓸 만한가**를 물었다는 점에서 의미가 있다.

## 저자들은 어떻게 접근했나

### 1. ASI 발견

연구팀은 TRANCO 상위 100개 웹 서비스에 테스트 계정을 만들고, 각 서비스에서 로그인 관련 정보를 제공하는 UI를 30분 이내 수동 탐색으로 찾았다. 그 결과 71개 서비스에서 200개의 ASI를 발견했고, 29개 서비스는 ASI 자체가 없었다.

이 수치 하나만으로도 이미 중요한 메시지가 나온다. 적지 않은 서비스가 사용자가 스스로 계정 침해를 확인할 창구를 아예 제공하지 않는다는 뜻이기 때문이다.

### 2. ASI 유형론 구축

발견한 인터페이스를 귀납적·연역적 코딩으로 분석해 7개 유형으로 분류했다.

| 유형 | 설명 |
| --- | --- |
| 보안 알림 | 계정 보안 변경 사항을 이메일·SMS로 알림 |
| 인증 알림 | 로그인 시 인증 코드가 포함된 알림 |
| 보안 로그 | 로그인 시도, 비밀번호 변경 등 보안 이벤트 기록 |
| 활동 로그 | 검색 기록, 구매 내역 등 비보안 활동 기록 |
| 기기 로그 | 계정에 접속한 기기 목록 |
| 감사 로그 | 관리자·개발자용 기술 이벤트 기록 |
| 세션 로그 | 로그인 세션 단위의 접속 기록 |

이 분류가 중요한 이유는, 우리가 보통 전부 "로그인 기록" 정도로 뭉뚱그려 생각하는 기능들이 실제로는 서로 다른 목적과 한계를 가진다는 점을 보여주기 때문이다.

### 3. 스푸핑 공격 테스트

저자들은 피해자 기기와 공격자 기기 두 대를 사용해 실제 스푸핑 시나리오를 재현했다. 공격자 기기에는 세 가지 조작을 적용했다.

- VPN과 SOCKS 프록시로 위치 정보 위조
- HTTP User-Agent 문자열 변조로 기기·브라우저 정보 위조
- 시스템 시계 조작으로 시간 정보 왜곡

이후 피해자 계정으로 로그인하고, ASI에 어떤 정보가 남는지 기록했다. 여기서 좋은 점은 논문이 단순히 설계 원칙을 말하는 데서 끝나지 않고, 실제로 **인터페이스가 공격자에게 얼마나 잘 속는지**를 검증했다는 점이다.

### 4. 정보 유용성 척도

저자들은 ASI가 남기는 정보가 얼마나 유용한지를 6단계 척도로 정리했다.

| 레벨 | 제공 정보 |
| --- | --- |
| 1 | 접근 정보 없음 |
| 2 | 날짜·시간만 |
| 3 | 위치 정보 |
| 4 | OS·브라우저 종류 |
| 5 | 버전 정보, 전체 User-Agent |
| 6 | 기기 고유 식별자 |

이 척도 덕분에 "로그는 있는데 별로 도움이 안 된다"는 막연한 느낌을 비교 가능한 기준으로 바꿀 수 있었다.

## 결과는 무엇을 보여줬나

### 1. ASI가 아예 없는 서비스가 적지 않았다

100개 중 29개 서비스는 ASI 자체가 없었다. booking.com, flickr.com, tiktok.com 등이 여기에 포함된다. 사용자가 스스로 계정 침해를 의심하고 확인할 수단이 사실상 없는 셈이다.

이 결과는 생각보다 무겁다. 보통 계정 보안은 인증 수단의 문제로만 다뤄지지만, 이 논문은 **탐지 인터페이스의 부재 자체가 보안 결함**일 수 있음을 보여준다.

### 2. ASI가 있어도 스푸핑에 쉽게 속았다

가장 강한 결과는 여기서 나온다. 스푸핑에 취약한 서비스가 41개였고, 비율로는 67.2%였다. 위치 정보는 70개 서비스 모두에서 VPN으로 위조 가능했고, 기기나 브라우저 정보도 38개 서비스에서 User-Agent 조작만으로 속아 넘어갔다. 스푸핑 이후 45개 서비스는 사실상 날짜와 시간 정도만 믿을 수 있는 수준으로 떨어졌다.

즉, 사용자가 "이건 다른 도시에서 로그인했으니 수상하다"거나 "이 기기는 내가 안 쓰는 브라우저니까 이상하다"라고 판단하려 해도, 그 정보 자체가 너무 쉽게 조작될 수 있다는 뜻이다.

### 3. 같은 서비스 안에서도 정보가 서로 맞지 않았다

하나의 서비스 안에 여러 ASI가 존재하더라도, 서로 다른 정보를 보여주는 경우가 흔했다. 세션 로그와 보안 로그가 서로 다른 기록을 표시하거나, 활성 세션 수가 실제와 일치하지 않는 사례도 있었다. 논문에서 언급된 twitter.com 사례가 대표적이다.

이 결과는 단순한 UI 혼란을 넘어선다. 사용자가 여러 화면을 비교해가며 침해 여부를 판단해야 할 때, **인터페이스 간 불일치 자체가 탐지를 방해하는 요소**가 된다.

### 4. 찾기 어려운 ASI는 사실상 없는 것과 비슷했다

ASI에 도달하기까지 평균 3.34번의 클릭이 필요했고, 더 복잡한 서비스도 있었다. 특정 서비스는 7단계 가까운 탐색이 필요했다. 계정 침해를 의심하는 사용자가 긴박한 상황에서 이런 경로를 끝까지 따라가기는 쉽지 않다.

보안 기능은 존재하는 것만으로 충분하지 않다. **필요한 순간에 곧바로 도달할 수 있어야** 의미가 있다.

### 5. 정보가 비어 있거나 설명이 부족한 경우도 많았다

논문은 xiaomi.com 사례처럼 두 기기로 로그인했는데도 기기 로그가 빈 채 남아 있는 경우를 보여준다. 또 일부 서비스는 왜 특정 항목이 기록되었는지, 세션과 기기의 차이가 무엇인지, 사용자가 어떤 조치를 해야 하는지 설명이 없었다.

결국 사용자는 보안 UI를 마주하더라도 그 정보가 **무엇을 의미하는지 해석할 단서조차 부족한 상황**에 놓인다.

## 이 논문이 남긴 개선 방향

저자들은 단순히 문제를 나열하는 데서 멈추지 않고 6가지 개선 원칙을 제안한다.

1. 완전한 로깅: 활성·비활성·로그아웃 세션 모두 기록하기
2. 높은 유용성: 기기 구분에 충분한 정보 제공하기
3. 명확한 용어: 세션과 기기 같은 개념을 혼용하지 않기
4. 동작 가시성: 왜 이 항목이 남았는지 사용자에게 설명하기
5. 효과적인 안내: 침해 징후와 대응 방법 함께 제공하기
6. 쉬운 접근성: ASI에 빠르게 도달할 수 있도록 경로를 단순화하기

흥미로운 점은 이 여섯 원칙을 모두 만족한 서비스가 단 하나도 없었다는 사실이다. 즉, 문제는 몇몇 예외적인 서비스의 실패가 아니라 **업계 전반의 설계 미성숙**에 가깝다.

## 이 연구의 한계는 무엇인가

이 논문도 한계를 분명히 인정한다.

- 영어 서비스 중심이라 다국어·지역 서비스로 일반화하기 어렵다.
- 모바일 앱은 포함되지 않았다.
- 수동 탐색 방식이라 ASI를 놓쳤을 가능성이 있다.
- 서비스당 30분 탐색 제한이 누락을 만들 수 있다.
- 상위 100개 서비스 중심이라 의료·금융 같은 민감 산업의 중소 서비스는 포함되지 않았다.

이 한계들은 동시에 후속 연구의 방향이기도 하다. 특히 모바일 앱 ASI와 규제 산업 서비스는 바로 이어서 확장하기 좋은 공백처럼 보인다.

## 내가 느낀 점

이 논문을 읽고 가장 먼저 남은 건 숫자였다. 100개 중 29개는 ASI가 없고, 존재하는 것 중 상당수는 스푸핑에 취약하다. 사용자가 마지막으로 기대는 "계정 보안 확인" 화면이 사실은 생각보다 훨씬 허술할 수 있다는 점을 실측으로 보여준다는 점에서 강한 경각심이 들었다.

또 하나 인상 깊었던 건 공시 과정이다. 연구팀이 58개 기관에 결과를 알렸지만, 상당수 답변은 "우리 범위 밖이다", "이미 MFA가 있으니 괜찮다", "물리적 접근이나 친밀한 관계를 전제한 공격은 대상이 아니다"에 가까웠다. 이 대목은 계정 탈취 이후의 **탐지와 회복** 문제가 기존 버그 바운티나 취약점 처리 문화 안에서 얼마나 주변화되어 있는지를 잘 보여준다.

개인적으로도 이 논문은 사용자의 습관을 다시 보게 만들었다. 나 역시 여러 서비스의 로그인 기록, 최근 기기, 보안 알림 화면을 대충 보고 넘긴 적이 많았다. "아마 내가 다른 기기에서 로그인한 거겠지" 하고 무심코 신뢰한 경우도 있었다. 그런데 이 논문을 읽고 나니 문제는 사용자의 부주의만이 아니라, **애초에 그 인터페이스가 충분히 신뢰할 수 있게 설계되지 않았을 수 있다**는 점이 더 선명하게 보였다.

- 스푸핑 테스트를 자동화해 더 넓은 서비스 집합을 주기적으로 점검하는 파이프라인 구성

### 연구나 프로젝트로 발전시킨다면

이 논문이 제시한 6가지 원칙을 정량 평가 지표로 바꿔, 서비스별 `ASI 보안 점수`를 산출하는 도구를 만들어보고 싶다. 설명 부재, 용어 혼용, 탐색 경로 복잡성 같은 항목은 UI 구조 분석과 NLP를 결합하면 어느 정도 반자동 평가가 가능할 수도 있다. 그렇게 되면 더 많은 서비스를 더 자주, 더 일관된 기준으로 비교할 수 있다.

## 정리

이 논문은 계정 보안을 "로그인 막기"의 문제에서 "침해를 사용자가 알아차릴 수 있게 돕기"의 문제로 확장해 보여준다. 인증 기술이 강해져도, 사용자가 믿는 보안 인터페이스가 불완전하고 쉽게 속는다면 탐지와 대응은 여전히 어렵다. 그래서 이 논문은 단순히 ASI를 조사한 논문이 아니라, **사용자에게 보안 신호를 어떻게 보여줄 것인가**라는 더 큰 질문을 던지는 연구로 읽혔다.
